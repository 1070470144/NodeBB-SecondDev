# Feature Specification: 剧本（JSON）功能

**Feature Branch**: `002-script-management`  
**Created**: 2026-03-04  
**Status**: Draft  
**Input**: User descriptions: "创建剧本功能，前端需要：剧本上传，剧本列表，管理端需要可以对用户上传的剧本进行管理，剧本本质上是一个json文件。包括但不限于以上剧本相关功能"; "论坛需要一个页签，可以展览所有用户上传的剧本，可以搜索剧本，根据热度和下载量等筛选"; "剧本页签所有功能的剧本要可以点赞，收藏，用户需要有我的收藏页签"

## User Scenarios & Testing *(mandatory)*

## Clarifications

### Session 2026-03-04

- Q: 论坛“剧本”页签要对哪些人可见？ → A: 可配置（默认所有访客可见）
- Q: “热度”排序/筛选的口径是什么？ → A: 热度 = 点赞与收藏等互动量
- Q: “下载量”的统计口径是什么？ → A: 同一用户对同一剧本终身最多计 1 次
- Q: 上传的“剧本 JSON”最小结构要求是什么？ → A: JSON 结构不限制，但必须填写标题/描述等元信息
- Q: 用户上传剧本后的“默认可见性”是什么？ → A: 由管理端配置决定默认值（可选私有/公开）

### User Story 1 - 上传剧本（JSON 文件）(Priority: P1)

作为已登录用户，我希望能上传一个剧本（JSON 文件），系统能校验其格式并保存，上传成功后我能在自己的剧本列表中看到它，以便后续使用与管理。

**Why this priority**: 上传是所有后续“列表/管理/使用”的前置条件，是最核心的价值交付。

**Independent Test**: 使用一个有效 JSON 文件完成上传，随后在“我的剧本列表”中可见该条目且元信息正确。

**Acceptance Scenarios**:

1. **Given** 用户已登录且具备上传权限，**When** 选择一个有效 JSON 文件并填写标题/描述等必填元信息后提交上传，**Then** 系统提示上传成功并创建对应剧本记录，且在列表中可见
2. **Given** 用户已登录，**When** 上传的文件不是有效 JSON 或缺少必填元信息（如标题/描述），**Then** 系统拒绝保存并给出可理解的错误提示
3. **Given** 用户已登录，**When** 上传超出允许大小/数量限制的文件，**Then** 系统拒绝并提示限制与解决方式（如减小文件/删除旧剧本）

---

### User Story 2 - 浏览与管理自己的剧本列表 (Priority: P2)

作为已登录用户，我希望能查看自己上传的剧本列表，并能进行常见管理操作（查看详情、更新、删除），以便维护我的剧本库。

**Why this priority**: 没有列表与基本管理，用户无法有效复用与维护已上传的剧本，上传价值难以体现。

**Independent Test**: 先上传 2 个剧本后，进入列表能看到它们，能完成一次更新与一次删除并在列表中反映。

**Acceptance Scenarios**:

1. **Given** 用户已上传至少一个剧本，**When** 打开剧本列表，**Then** 列表展示剧本关键元信息并支持分页/筛选（若数据量较大）
2. **Given** 用户拥有某个剧本，**When** 对该剧本执行更新（重新上传新 JSON 或修改元信息），**Then** 更新生效且保留必要的审计信息（如更新时间、更新者）
3. **Given** 用户拥有某个剧本，**When** 删除该剧本并确认，**Then** 剧本从用户列表中消失且不会再被普通用户访问

---

### User Story 3 - 管理端审核与管理用户剧本 (Priority: P3)

作为管理端人员（管理员/版主/运营），我希望能查看全站用户上传的剧本，并对其进行管理（搜索、查看、下架/封禁、删除、恢复等），以便控制内容质量与合规风险。

**Why this priority**: 这是平台治理能力，虽然不影响最小可用上传能力，但对长期运营与风险控制关键。

**Independent Test**: 准备多个用户上传的剧本后，管理端能按条件检索，能对某条剧本执行一次“下架/封禁”并在用户侧体现为不可用。

**Acceptance Scenarios**:

1. **Given** 管理员已登录且具备管理权限，**When** 打开管理端剧本列表，**Then** 能看到全站剧本并支持按上传者/状态/时间等条件检索
2. **Given** 管理员选择某个剧本，**When** 将其标记为不可用（如下架/封禁），**Then** 该剧本在用户侧不可被正常访问/使用，且能向上传者展示原因与申诉入口（若平台有申诉机制）
3. **Given** 管理员对某剧本执行删除/恢复（若支持），**When** 操作完成，**Then** 系统记录操作者与原因，并在列表中正确反映状态变化

---

### User Story 4 - 论坛剧本展示与筛选页签 (Priority: P3)

作为论坛的访问用户（默认所有访客可见，且可由管理端配置可见范围），我希望论坛有一个专门的剧本页签，可以浏览所有对外可见的用户剧本，并能通过搜索和按热度、下载量等条件筛选/排序，从而更容易发现优质剧本。

**Why this priority**: 该页签提升剧本的曝光与分发效率，有助于提高剧本被实际使用的价值，但依赖基础上传/列表能力之后才能发挥作用。

**Independent Test**: 在已有多名用户上传剧本的前提下，通过论坛页签完成一次“按关键词搜索 + 按下载量排序”的浏览，并成功定位到目标剧本。

**Acceptance Scenarios**:

1. **Given** 系统中存在若干已设置为对外可见（公开）的剧本，**When** 用户打开论坛中的“剧本”页签，**Then** 页签展示这些剧本的列表及关键元信息，并提供基础分页能力
2. **Given** 用户处于“剧本”页签，**When** 在搜索框输入关键词并提交，**Then** 列表根据标题/描述等字段进行匹配，只展示符合条件的剧本
3. **Given** 用户处于“剧本”页签，**When** 选择按热度或下载量排序，**Then** 列表按所选条件重新排序，其中热度以收藏/点赞等互动量为口径，并在视觉上清晰体现当前排序方式

---

### User Story 5 - 点赞、收藏与“我的收藏”页签 (Priority: P2)

作为论坛的已登录用户，我希望能对剧本进行点赞与收藏，并在“我的收藏”页签中集中查看我收藏过的剧本，以便快速回访优质内容与持续使用。

**Why this priority**: 点赞/收藏直接支撑“热度”与内容分发质量，也为用户提供内容沉淀与回访路径，提升剧本生态的活跃度与粘性。

**Independent Test**: 登录后对某剧本完成一次点赞与收藏操作，随后在“我的收藏”页签中能找到该剧本；取消收藏后该剧本从收藏列表移除。

**Acceptance Scenarios**:

1. **Given** 用户已登录且剧本为对外可见（公开），**When** 用户对该剧本点击“点赞”，**Then** 系统记录该用户的点赞状态并更新该剧本的点赞数（同一用户对同一剧本最多计 1 次）
2. **Given** 用户已登录且剧本为对外可见（公开），**When** 用户对该剧本点击“收藏”，**Then** 系统记录该用户的收藏状态并更新该剧本的收藏数（同一用户对同一剧本最多计 1 次）
3. **Given** 用户已登录且已收藏至少一个剧本，**When** 打开“我的收藏”页签，**Then** 系统展示该用户收藏的剧本列表，并支持从列表进入剧本详情
4. **Given** 用户已登录且已收藏某剧本，**When** 用户取消收藏，**Then** “我的收藏”页签不再展示该剧本，且该剧本收藏数相应更新

---

### Edge Cases

- 上传文件为空、编码异常、或包含无法解析的字符时如何处理？
- 同名/重复上传（内容完全一致或高度相似）时如何提示与去重（若有）？
- 用户删除剧本后，若其他地方引用该剧本（如被收藏/被引用），应如何表现？
- 管理员下架后，用户访问旧链接/旧入口时应返回什么反馈？
- 并发更新：用户同时在两个窗口更新同一个剧本时，如何避免覆盖与数据丢失？
- 在“剧本”页签中，当筛选条件过多且无匹配结果时，如何提示用户调整条件？
- 下载量或互动量出现疑似刷量时，治理与展示应如何处理（如限制计数、回滚、下架）？
- 未登录用户看到点赞/收藏入口时，应该如何引导登录？
- 用户对同一剧本反复点赞/取消点赞、收藏/取消收藏时，计数与状态如何保持一致？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 允许已登录用户上传剧本文件，且剧本文件以 JSON 为本质内容载体
- **FR-002**: 系统 MUST 在保存前对上传内容进行基础校验（至少包含“可被解析为 JSON”）并校验必填元信息（例如标题/描述）
- **FR-003**: 系统 MUST 为每个剧本保存可供列表展示的元信息（例如：标题、描述、上传者、创建时间、更新时间、状态）
- **FR-004**: 用户 MUST 只能查看与管理自己上传的剧本（除非该剧本被明确设置为对外可见（公开）或被授予指定可见范围）
- **FR-005**: 系统 MUST 提供用户侧“剧本列表”，支持在数据量增大时依然可用的浏览方式（例如分页/筛选/排序）
- **FR-006**: 系统 MUST 支持用户对自己的剧本执行更新（重新上传新 JSON 或更新元信息）与删除
- **FR-007**: 系统 MUST 防止越权访问：用户不得读取/修改/删除不属于自己的剧本
- **FR-008**: 系统 MUST 提供管理端对全站剧本的查看与检索能力（按上传者、状态、时间范围、关键词等）
- **FR-009**: 管理端 MUST 能对剧本进行治理操作（例如：下架/封禁/删除/恢复），并能记录操作原因与操作者
- **FR-010**: 当剧本因管理端治理变为不可用时，系统 MUST 在用户侧以可理解方式呈现不可用状态与原因（在不泄露敏感信息前提下）
- **FR-011**: 系统 MUST 对关键操作保留审计信息（至少：上传、更新、删除、下架/封禁、恢复），以便追溯
- **FR-012**: 系统 MUST 对上传大小、频率、数量等设置合理限制，并在触发限制时给出可操作的提示
- **FR-013**: 系统 MUST 在论坛中提供一个剧本页签/入口，用于展示符合对外可见条件的用户剧本列表；该页签的可见范围 MUST 支持由管理端配置，且默认对所有访客可见
- **FR-014**: 在论坛剧本页签中，系统 MUST 提供基于标题、描述或其他关键字段的搜索能力，帮助用户快速定位目标剧本
- **FR-015**: 在论坛剧本页签中，系统 MUST 支持按热度与下载量等排序/筛选条件对剧本列表进行重新排列或过滤；其中热度口径为点赞量与收藏量等互动量
- **FR-016**: 系统 MUST 以一致口径统计并展示下载量用于排序/筛选；下载量口径为“同一用户对同一剧本终身最多计 1 次”
- **FR-017**: 系统 MUST 支持由管理端配置“新上传剧本的默认可见性”（私有或公开），并确保论坛页签仅展示对外可见（公开）的剧本
- **FR-018**: 已登录用户 MUST 能对对外可见（公开）的剧本执行点赞与取消点赞；系统 MUST 确保同一用户对同一剧本最多计 1 次点赞
- **FR-019**: 已登录用户 MUST 能对对外可见（公开）的剧本执行收藏与取消收藏；系统 MUST 确保同一用户对同一剧本最多计 1 次收藏
- **FR-020**: 系统 MUST 为每个对外可见（公开）的剧本展示可理解的互动信息（至少包含点赞数、收藏数与用户自身状态）
- **FR-021**: 系统 MUST 提供“我的收藏”页签/入口，仅对已登录用户可用，用于展示该用户收藏的剧本列表，并支持取消收藏

### Key Entities *(include if feature involves data)*

- **Script（剧本）**: 用户上传的 JSON 剧本资源；关键属性包括唯一标识、标题/描述（由用户填写的元信息）、上传者、创建/更新时间、当前状态（可用/不可用/删除等）、内容（JSON，不限制结构）、可见性范围（如私有/公开/指定可见）、下载量（同一用户终身最多计 1 次）、点赞量、收藏量
- **Script Interaction（剧本互动）**: 用户与剧本的互动关系；关键属性包括剧本标识、用户标识、互动类型（点赞/收藏）、创建时间、状态（有效/取消）
- **Script Moderation Record（治理记录）**: 管理端对剧本执行治理操作的记录；关键属性包括剧本标识、动作类型（下架/封禁/恢复/删除等）、原因、操作者、操作时间

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 95% 的有效剧本上传在 30 秒内完成（以常见网络环境与文件大小上限为前提）
- **SC-002**: 当用户拥有 1,000 个剧本时，用户侧列表页面首屏可在 2 秒内可用（包含基本信息展示）
- **SC-003**: 在可用性测试中，至少 90% 的目标用户能在首次尝试中完成“上传一个有效 JSON 并在列表中找到它”
- **SC-004**: 上线后 4 周内，与“剧本上传失败/无法找到剧本/越权访问”相关的支持工单数量较上线前同类问题下降 30%
- **SC-005**: 上线后 4 周内，通过论坛剧本页签产生的剧本浏览/下载行为占剧本总浏览/下载行为的比例达到预期指标（例如 50%），且用户在可用性测试中对该页签的可发现性与易用性评分不低于预设门槛
- **SC-006**: 在可用性测试中，至少 90% 的已登录目标用户能在 1 分钟内完成“收藏一个剧本并在我的收藏中找到它”

## Assumptions

- 用户与管理端人员都已存在可用的登录与权限体系；本功能只在此体系之上增加“剧本资源”的权限约束与管理能力
- 若未来需要对剧本 JSON 结构做更强约束，将以不破坏既有数据为前提逐步引入，并提供对旧数据的兼容策略
